## 开发循环流程（每个任务必须严格遵循）

**核心原则：每轮迭代必须完整执行所有步骤，Code Review 必须全面扫描（不是只验证上轮修复）**

### 强制执行流程

```
开始任务

    ▼

│  2.4 确保每个测试都有明确的目的和断言                              │
│  2.5 运行全部测试，确认通过                                       │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│ 第 3 阶段：全面 Code Review（必须全面扫描，不是增量验证）          │

│                                                                  │

│                                                                  │

    │
```

### 测试要求（强制）

| 测试类型 | 要求 | 示例 |
|---------|------|------|
| **单元测试** | 核心逻辑必须覆盖 | `test_count_tokens_tiktoken_basic` |
| **E2E 测试** | 必须使用真实 API，不能用 mock | `test_token_count_accuracy_vs_api_response` |
| **边界测试** | 异常情况和边界条件 | `test_empty_input_returns_zero` |

⚠️ **禁止**：只写纯逻辑单元测试而不写 e2e 测试

### Code Review 执行规范

**每轮 Code Review 必须：**

1. **全面扫描所有修改过的文件**（不是只看上轮改的地方）
2. **使用统一的检查清单**（见上方第 3 阶段）
3. **记录所有发现的问题**（包括非本轮引入的问题）
4. **区分问题严重性**：
   - 🔴 **严重**：必须立即修复（逻辑错误、安全漏洞、数据丢失风险）
   - 🟡 **中等**：应该修复（性能问题、代码质量问题）
   - 🟢 **低**：可以改进（风格问题、文档补充）

**禁止的做法：**
- ❌ 只验证上轮修复是否正确
- ❌ 跳过未修改的文件
- ❌ 忽略"看起来没问题"的代码

### 终止条件（必须全部满足）

1. ✅ 所有 TODO 项已完成
2. ✅ 测试通过率 ≥ 95%（允许 5% 的 flaky 测试）
3. ✅ 连续 2-3 轮迭代无重大漏洞
4. ✅ 无新增的安全或性能问题

## Code Review 要求（补充说明）

**⚠️ 详细流程已在上方「开发循环流程」中定义，此处为补充说明**

### 审查原则
- **项目定位**: 此项目为个人实践项目，非生产级系统
- **客观评价**: 保持客观，不夸大问题严重性
- **务实建议**: 给出可操作的改进建议，而非理论性批评
- **安全优先**: 重点关注安全漏洞和数据泄露风险

### 审查限制
- **不接入服务器**: 不要求部署到生产环境
- **不接入外部系统**: 不要求集成第三方服务
- **个人项目标准**: 按照个人项目标准评估，非企业级标准

### 关键教训

1. **测试必须包含 e2e**：不能只写单元测试，必须有真实 API 调用的端到端测试
2. **测试精简是必要步骤**：每轮迭代后必须检查测试是否可精简
3. **Code Review 必须全面**：每轮 review 都是全面扫描，不是只看上轮改的地方
4. **流程必须强制执行**：靠"提醒"不如靠"流程+检查清单"
